Week 6: Data Engineering & The Modern Data Stack

Module Introduction

Welcome to the foundation of all data-driven decision making. While data analysis often receives the spotlight, none of it would be possible without the critical work of data engineering. This module explores the behind-the-scenes infrastructure that enables businesses to collect, store, process, and transform raw data into actionable insights. You'll learn how data moves from countless sources to become the clean, structured information that powers dashboards, reports, and machine learning models.

Module Learning Objective

By the end of this week, students will understand that before analysis can happen, data must be ingested, stored, and transformed. This module introduces the foundational concepts of data engineering that make BI and analytics possible.

6.1: The Data Pipeline Concept: ETL vs. ELT

1. The Data Journey: From Source to Insight

Data doesn't magically appear in dashboards and reports. It undergoes a sophisticated journey through multiple stages, each with specific challenges and requirements.

The Complete Data Pathway

  • Sources: The origins of data, which can include:
    • Databases (SQL, NoSQL)
    • APIs (application programming interfaces)
    • File systems (CSV, JSON, Parquet)
    • Streaming data (clickstreams, IoT sensors)
    • Third-party data providers
  • Ingestion: The process of collecting and importing data from sources
    • Batch processing: Scheduled data transfers (hourly, daily)
    • Real-time processing: Continuous data collection
    • Change data capture: Identifying and capturing data changes
  • Storage: Where data resides during processing
    • Data lakes: Raw, unstructured data storage
    • Data warehouses: Structured, processed data
    • Data marts: Department-specific data subsets
  • Transformation: Converting raw data into analysis-ready formats
    • Cleaning: Removing errors and inconsistencies
    • Standardization: Ensuring consistent formats
    • Enrichment: Augmenting with additional data sources
    • Aggregation: Summarizing detailed data
  • Analytics: The process of deriving insights
    • Business intelligence: Dashboards and reports
    • Data science: Statistical analysis and modeling
    • Machine learning: Predictive analytics
  • Insights: Actionable information delivered to decision-makers
    • Visualization: Charts, graphs, and dashboards
    • Alerts: Automated notifications of significant findings
    • Recommendations: Data-driven suggestions for action

Pipeline Reliability Considerations

  • Data Quality: Ensuring accuracy, completeness, and consistency
  • Latency: Balancing speed with comprehensiveness
  • Scalability: Handling increasing data volumes efficiently
  • Monitoring: Tracking pipeline health and performance
2. ETL (Extract, Transform, Load)

ETL represents the traditional approach to data processing, where transformation occurs before loading into the target system.

The ETL Process Breakdown

Extract
  • Pulling data from source systems
  • Handling different data formats and structures
  • Managing API rate limits and connection constraints
  • Implementing incremental extraction to avoid full reloads
Transform
  • Data Cleaning: Handling missing values, standardizing formats, correcting errors
  • Business Rules Application: Calculating metrics, applying logic, creating derived fields
  • Data Validation: Ensuring quality and consistency
  • Performance Optimization: Aggregating, filtering, and structuring for efficiency
Load
  • Inserting transformed data into the target data warehouse
  • Managing schemas and table structures
  • Handling updates, inserts, and deletions
  • Maintaining historical data (slowly changing dimensions)

ETL Advantages

  • Data Quality: Transformation before loading ensures clean, consistent data
  • Compliance: Sensitive data can be anonymized or encrypted before storage
  • Performance: Pre-aggregated data enables faster query performance
  • Cost Control: Storage costs are optimized by storing only processed data

ETL Challenges

  • Rigidity: Schema changes require pipeline modifications
  • Latency: Transformation step can delay data availability
  • Resource Intensive: Requires significant processing power before loading
  • Limited Raw Data Access: Original data may be lost during transformation

Real-World ETL Example: Financial Services Compliance

A bank must generate daily regulatory reports showing transaction patterns. The ETL process:

  1. Extracts transaction data from multiple banking systems
  2. Transforms data by anonymizing customer information, converting currencies, and flagging suspicious patterns
  3. Loads the compliant, standardized data into a regulatory reporting database
  4. Analysts run predefined reports without accessing raw transaction data
3. ELT (Extract, Load, Transform)

ELT represents the modern approach to data processing, leveraging the power of cloud data platforms to transform data after loading.

The ELT Process Breakdown

Extract
  • Similar to ETL, but often with less preprocessing
  • May include lightweight parsing or compression
  • Focus on rapid data movement to the target platform
Load
  • Storing raw, largely unprocessed data in the target system
  • Using scalable cloud storage solutions
  • Maintaining data in its original form for future flexibility
  • Implementing efficient loading patterns for large volumes
Transform
  • Performing transformations within the data warehouse
  • Using SQL or warehouse-specific processing capabilities
  • Creating transformed views rather than physical tables
  • Enabling multiple transformation versions for different use cases

ELT Advantages

  • Flexibility: Raw data is available for unexpected analysis needs
  • Speed: Data is available sooner since transformation happens after loading
  • Scalability: Cloud data platforms handle transformation at scale
  • Iterative Development: Transformations can be modified without reloading data

ELT Challenges

  • Data Governance: Raw data requires careful access controls
  • Storage Costs: Storing raw data can be more expensive
  • Skill Requirements: Requires strong SQL and data modeling expertise
  • Performance Management: Poorly designed transformations can impact warehouse performance

Real-World ELT Example: E-commerce Customer Analysis

An online retailer wants to analyze customer behavior across multiple touchpoints:

  1. Extracts raw clickstream, purchase, and customer service data
  2. Loads all raw data into a cloud data warehouse (Snowflake/BigQuery)
  3. Data scientists transform data within the warehouse to create customer journey maps
  4. Different teams create their own transformed views for specific analysis needs
  5. Raw data remains available for unexpected investigation needs
4. When to Use ETL vs. ELT

ETL is Preferable When

  • Regulatory Compliance: Industries with strict data governance requirements (healthcare, finance)
  • Established Requirements: Well-defined reporting needs with consistent formats
  • Performance Critical: Applications requiring optimized query performance
  • Sensitive Data: Scenarios where raw data shouldn't be stored for security reasons
  • Limited Storage: Environments with significant storage constraints

ELT is Preferable When

  • Exploratory Analysis: Situations requiring flexibility for unexpected questions
  • Cloud Infrastructure: Organizations using modern cloud data platforms
  • Rapid Implementation: Projects needing quick time-to-value for data availability
  • Multiple Use Cases: Data serving diverse analytical requirements
  • Data Science Needs: Scenarios where raw data is valuable for modeling

Hybrid Approaches

Many organizations implement a hybrid strategy:

  • ETL for sensitive, regulated, or high-performance data
  • ELT for exploratory, innovation, or raw data analysis
  • Creating a "medallion architecture" with bronze (raw), silver (cleaned), and gold (enriched) layers

Decision Framework

Consider these factors when choosing between ETL and ELT:

  1. Data Sensitivity: How strictly must raw data be controlled?
  2. Analysis Patterns: Are reporting needs predictable or exploratory?
  3. Technical Environment: What platforms and tools are available?
  4. Team Skills: What expertise exists for transformation logic?
  5. Cost Structure: How does your organization balance storage vs. processing costs?
  6. Compliance Requirements: What regulations govern your data handling?

Implementation Considerations

  • Tool Selection: Choose tools that support your preferred approach (ETL tools: Informatica, Talend; ELT tools: dbt, Dataform)
  • Monitoring: Implement comprehensive monitoring regardless of approach
  • Documentation: Maintain clear documentation of data lineage and transformation logic
  • Testing: Establish robust testing procedures for transformation logic
  • Version Control: Manage transformation code with proper version control systems

ETL vs. ELT Explained

Key Takeaway

The choice between ETL and ELT represents a fundamental architectural decision in data engineering. ETL offers control, compliance, and performance for well-understood data needs, while ELT provides flexibility, speed, and adaptability for exploratory analysis. Modern data stacks often incorporate elements of both approaches, creating layered architectures that balance governance with flexibility. Understanding these patterns is essential for designing data systems that effectively support organizational analytics needs.
